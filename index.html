Quais são as vantagens da computação sem servidor?

A computação sem servidor permite que os desenvolvedores criem aplicativos sem a dor de cabeça de gerenciar a infraestrutura.
Mais especialmente, permite que eles escrevam em código sem servidor sem precisar:

  • Provisionar um servidor
  • Garanta sua funcionalidade
  • Criar ambientes de teste em um servidor
  • Manter o tempo de atividade do servidor

Realmente sem Servidor?

Não quer dizer que não existe um servidor, mas que não preocupamos em configurar esses servidores. Ou seja, só preocupamos com nosso código.

Benefícios:

  • Custos mais baixos - A computação sem servidor geralmente é muito econômica, pois os provedores tradicionais de serviços de back-end em nuvem (alocação de servidor) geralmente fazem com que o usuário pague por espaço não utilizado ou tempo de CPU ocioso.
  • Escalabilidade simplificada - Os desenvolvedores que usam arquitetura sem servidor não precisam se preocupar com políticas para escalar seu código. O fornecedor sem servidor lida com todo o dimensionamento sob demanda.
  • Código de back-end simplificado - Com o FaaS, os desenvolvedores podem criar funções simples que executam de forma independente uma única finalidade, como fazer uma chamada de API.
  • Resposta mais rápida - A arquitetura sem servidor pode reduzir significativamente o tempo de lançamento no mercado. Em vez de precisar de um processo de implantação complicado para lançar correções de bugs e novos recursos, os desenvolvedores podem adicionar e modificar o código aos poucos.

Functions-as-a-Service (FaaS)
(serverless function)

Uma função serverless é uma função programática escrita por um desenvolvedor de software para um único propósito. Em seguida, é hospedado e mantido em infraestrutura por empresas de computação em nuvem. Essas empresas cuidam da manutenção e execução do código para que os desenvolvedores possam implantar novos códigos com mais rapidez e facilidade.

Back-end como serviço (BaaS)

Permite que os desenvolvedores terceirizam a administração das funções de back-end.

Plataforma como serviço (PaaS)

Permite desenvolver e implantar produtos no ambiente de nuvem.

Infraestrutura como serviço (IaaS)

Permite desenvolver e implantar produtos no ambiente de nuvem.

Gateway de API

Um serviço totalmente gerenciado que faz com que os desenvolvedores criem, publiquem, mantenham, monitorem e protejam APIs facilmente em qualquer escala. O API Gateway pode ser usado para criar APIs Restful e APIs WebSocket que possuem aplicativos de comunicação bidirecional em tempo real. É uma porta de entrada para aplicativos acessarem dados, lógica de negócios ou funcionalidade de seus serviços de back-end.

AWS Lambda

Um serviço de computação sem servidor que executa código como resposta a eventos e cuida automaticamente dos recursos inferiores. Ele executa código em infraestrutura de computação de alta disponibilidade e realiza toda a administração dos recursos de computação. Alguns exemplos são solicitações HTTP via Amazon API Gateway, alterações em objetos no S3 e muitos outros.

DynamoDB

Um banco de dados do Amazon AWS Web Services que é totalmente gerenciado, sem servidor, banco de dados NoSQL de valor-chave, possui aplicativos de alto desempenho em qualquer escala. Possui os seguintes benefícios, como segurança integrada, backups contínuos, replicação multirregional automatizada, armazenamento em cache na memória e ferramentas de exportação de dados.

O que é AWS CloudFormation?

AWS CloudFormation é um serviço de infraestrutura como código (Infrastructure as Code IaC). Com ele podemos economizar tempo com gerenciamento de recursos e focar em desenvolvimento de regras de negocio. Tudo o que você precisa fazer é criar um modelo que descreva todos os recursos da AWS. Em seguida, o CloudFormation cuida do provisionamento e da configuração desses recursos.

O que é AWS SAM?

AWS SAM significa modelo de aplicações sem servidor (Serverless Application Model). E, é uma estrutura como código (Infrastructure as Code IaC), open source. Com o SAM podemos criar estruturas serverles, como: Lambda, Fargate, API Gateway, SNS, SQS, S3, DynamoDB e muito mais.

O que é AWS SAM?

Com ele podemos economizar tempo com gerenciamento de recursos e focar em desenvolvimento de regras de negocio. Tudo o que você precisa fazer é criar um modelo que descreva todos os recursos da AWS. Em seguida, o CloudFormation cuida do provisionamento e da configuração desses recursos.

Para usar o SAM, precisamos:
  • De permissões para aplicar e implantar alterações. Por exemplo, a CLI do AWS SAM faz chamadas para o Amazon S3 e AWS CloudFormation. Portanto, precisa de permissões para usar esses serviços.
  • Um modelo SAM que inclui todas as definições para sua infraestrutura sem servidor. Os modelos SAM usam uma sintaxe abreviada. Isso torna o modelo simples e clean.
  • LPara instalar o command-line interface (CLI), precisamos definir as credenciais da AWS para que ele possa fazer chamadas para os serviços da AWS. A CLI do SAM fornece ferramentas para o desenvolvedor criar e implantar as aplicações.
O que é AWS Lambda?

Lambda é um serviço de computação sem servidor e de alta disponibilidade. Assim, ele permite que você execute código sem provisionar ou gerenciar servidores, não precisando se preocupar com infraestrutura, administração, manutenção do sistema, provisionamento de capacidade, dimensionamento automático e registro em log.

Os benefícios do uso de AWS SAM, incluem:
  • Organizar componentes relacionados um único modelo SAM.
  • Usar a command-line interface (CLI) para criar, testar, depurar e implantar e deletar as aplicações SAM.
  • Tem integração com várias IDEs. Isso inclui AWS Cloud9, Visual Studio Code e muito mais. Você também pode usá-lo para compilar com o AWS CodeBuild. A implantação é facilitada com AWS CodeDeploy e Jenkins.

Seções Opcionais e Obrigatórias:

ver cada um, abaixo:

Transform (Obrigatório)

Primeiro, a seção de transformação especifica as macros usadas para processar seu modelo. Você deve incluir esta seção com um valor de “AWS::Serverless-2016-10-31”. Outras transformações são possíveis, mas só são relevantes para o AWS CloudFormation.

Globals (Opcional):

Esta seção é exclusiva do AWS SAM. Ele contém informações padrão para todas as suas funções serverless, APIs e tabelas simples. Assim, ajuda a evitar a duplicação de informações para cada recurso. Por exemplo, o código abaixo garantirá que todas as funções usem o runtime python3. Ele também define um tempo limite padrão: 

Globals:
Function:
  Runtime: python3
  Timeout: 180
  Handler: index.handler
  Environment:
	Variables:
  TABLE_NAME: data-table

Description (Opcional)

A descrição é uma string que descreve o modelo na forma de comentários. Por exemplo: 

Description: >
 Here are some
 details about
 the template.

Metadata (Opcional)

A seção de metadados inclui objetos que fornecem informações extras sobre o modelo. Aqui o AWS CloudFormation adiciona informações usadas pela interface GUI. Por exemplo: 

Metadata:
Instances:
  Description: "Information about the instances"
Databases: 
  Description: "Information about the databases"

Parameters (Opcional)

A seção de parâmetros contém valores a serem passados ​​para seu modelo. Você também pode substituir valores usando o parâmetro: “parameter-overrides” ao implantar. Por exemplo: 

Parameters:
InstanceTypeParameter:
  Type: String
  Default: t2.micro
  AllowedValues:
	- t2.micro
	- m1.small
	- m1.large
  Description: Enter t2.micro, m1.small, or m1.large. Default is t2.micro.

Mappings (Opcional)

Mapeamentos são como uma tabela de pesquisa. Você pode usá-lo para corresponder uma chave a um valor correspondente. Por exemplo, a função Fn::FindInMap faz a pesquisa para você nas seções Recursos e Saídas. O exemplo de mapeamentos abaixo é bastante útil, pois permite especificar um ID de AMI de instância do EC2 diferente para cada região. Esse tipo de pesquisa é necessário, pois os IDs de AMI nem sempre são os mesmos em todas as regiões. Por exemplo: 

Mappings: 
  RegionMap: 
    us-east-1: 
   	  "HVM64": "ami-0ff8a91507f77f867"
	us-west-1: 
	  "HVM64": "ami-0bdb828fd58c52235"
	eu-west-1: 
	  "HVM64": "ami-047bb4163c506cd98"
	ap-southeast-1: 
	  "HVM64": "ami-08569b978cc4dfa10"
	ap-northeast-1: 
	  "HVM64": "ami-06cd52961ce9f0d85"

FindInMap (Opcional)

Por fim, o exemplo abaixo mostra como você pode usar a função FindInMap. Como você pode ver abaixo, o recurso abaixo procura um valor na seção do mapa da região. Por exemplo:

Conditions (Opcional)

A seção de condições permite que você crie um recurso condicionalmente. Isso é útil quando você deseja executar ações em recursos específicos. Além disso, você pode usar condições para evitar a execução de ações, como recriar o armazenamento de dados. Por exemplo: 

Conditions:
  CreateProdResources: !Equals 
    - !Ref EnvType
    - prod

Conditions (Opcional)

O exemplo abaixo mostra como usar uma condição. Por exemplo, a condição CreateProdResources só implantará EC2Instance se for verdadeira. Por exemplo: 

							AWSTemplateFormatVersion: 2010-09-09
Parameters:
  EnvType:
    Description: Environment type.
    Default: test
    Type: String
    AllowedValues:
      - prod
      - test
    ConstraintDescription: must specify prod or test.
Conditions:
  CreateProdResources: !Equals 
    - !Ref EnvType
    - prod
Resources:
  EC2Instance:
    Type: 'AWS::EC2::Instance'
    Properties:
      ImageId: ami-0ff8a91507f77f867
  MountPoint:
    Type: 'AWS::EC2::VolumeAttachment'
    Condition: CreateProdResources
    Properties:
      InstanceId: !Ref EC2Instance
      VolumeId: !Ref NewVolume
      Device: /dev/sdh
  NewVolume:
    Type: 'AWS::EC2::Volume'
    Condition: CreateProdResources
    Properties:
      Size: 100
      AvailabilityZone: !GetAtt 
        - EC2Instance
        - AvailabilityZone

Resources (Opcional)

A seção de recursos contém informações sobre os recursos que você deseja implantar. A definição de recurso muda de acordo com o tipo de recurso. Leia a referência de tipos de recursos e propriedades da AWS para obter mais informações. Em seguida, conforme o exemplo neste tutorial do AWS SAM: 

							Resources:

        - x86_64

            Path: /hello
